# Repository variation policy

The shared contract is fixed. Repository profiles customize only facts that
must differ because the repository has a different product, topology, runtime,
provider, release model, or lifecycle phase.

## Required profile fields

A profile names:

- repository purpose, target identity source, and current lifecycle phase;
- docs router and semantic owner paths;
- app, package, generated-reference, public-doc, runbook, proof, evidence,
  archive, SPEC, plan, skill, and agent-guidance locations;
- exact focused and closeout commands;
- representative jobs and critical-journey inventory;
- important trust, boundary, provider, release, deployment, and rollback facts;
- repository-specific exclusions and accepted non-claims; and
- qualified exceptions with rationale, owner, evidence, review trigger, and
  retirement condition.

## Fixed fields

A profile must not redefine:

- invariant meanings or finding priorities;
- truth-layer precedence or document classes;
- capability, identity, authority, approval, or mutation boundaries;
- accepted/rejected/deferred/optional finding states;
- claim-to-proof and artifact-identity requirements;
- required impact-ledger categories;
- failed-work retention;
- feedback promotion and automation admission;
- proportional stopping; or
- the rule that local commands and provider facts come from the repository and
  current external readback.

## Variation test

Accept a variation only when:

1. a concrete repository job or boundary requires it;
2. the default would be false or materially misleading;
3. the profile names the narrower replacement;
4. evidence and an owner support it;
5. downstream routes and verification remain coherent; and
6. a review or retirement trigger is recorded.

Convenience, historical habit, or an existing contradiction is not a qualified
variation.
